# Design Notes

Rationale for the design decisions behind slim-webp-enc — a pure-TypeScript lossy WebP thumbnail encoder with alpha. What the library does and how to use it is in the [README](README.md); this document records *why* it is built the way it is.

## Architecture

A lossy WebP-with-alpha file is:

```
RIFF ("WEBP")
└── VP8X chunk        — extended-format header, alpha flag, canvas size
└── ALPH chunk        — alpha plane (prediction-filtered, coded as a headerless VP8L stream)
└── "VP8 " chunk      — VP8 intra keyframe (Y'CbCr 4:2:0)
```

Key simplifications in the VP8 core: fixed quality→QP mapping (no rate control), single segment, rate-distortion mode selection over the four 16×16 modes, the ten 4×4 B_PRED sub-modes, and the four chroma modes (libwebp `method=3` style, single-pass), adaptive token probabilities, loop-filter strength derived from QP. The mini-VP8L coder used for alpha (and for the palette-lossless path) skips color transforms, color cache, and subresolution entropy images — it keeps only the prediction transform, one set of canonical Huffman codes, and greedy LZ77 over a small set of candidate distances.

Out of scope by design: animation, ICC/EXIF/XMP metadata, general-purpose lossless RGB (only the ≤ 256-color palette subset), decode support, SIMD, multi-segment quantization, multi-pass / target-size rate control, sharp YUV.

## Key decisions

| Decision | Choice | Rationale |
|---|---|---|
| Alpha compression | Semi-lossy level reduction (default **32** levels since 2026-07-11, was 16; 0/255 pinned exactly) + mini-VP8L | The small alphabet gives tightly clustered residuals → short Huffman codes and long LZ77 matches. 16 was chosen as invisible at thumbnail scale, but banding on slow alpha gradients at larger sizes is far more visible than the payload delta (measured: 16→32 costs ~7 points of corpus total vs libwebp, which itself ships *lossless* alpha by default — 32 is still cheaper than Chromium's behavior). A welcome side effect: at 32 the quantized candidates compress worse, so the lossless plane wins the race outright on many smooth gradients — zero banding, pixel-exact |
| Smooth-gradient alpha planes | One rescue candidate: a **1-D 8-phase dither aligned with the gradient axis**, offered only when the mean profile across the perpendicular axis stays within half a quantization step; LZ77 candidate distances widened to {1–4, w±1, w, 2w, 4w} (2026-07-11) | 2-D ordered dithering defeats LZ77 outright on full-range gradient planes (worst case 6.4× libwebp's file size), and distance-{1, w}-only matching missed near-periodic structure — 4·w is the Bayer period. A pattern varying only along the gradient axis leaves rows (or columns) identical, so it compresses like an undithered plane while still sub-stepping the bands — and being 1-D it affords 8 phases instead of the Bayer 4. The perpendicular-flatness gate (profiles computed from row/column *means*, which cancel pixel noise) keeps it off radial/diagonal fades, where it would leave banding it can't fix. An **undithered** candidate was tried with a >25%-win margin and reverted the same day: smooth gradients are exactly where it compresses best *and* bands worst, so any size margin strips dithering precisely where it matters (visible on the demo's alpha-radial). Dithering is a quality contract, not a size heuristic — `alphaDither: 0` is the explicit opt-out. Worst case: 6.4× → 1.6× with dithering kept |
| Sub-visible-step dither rule | Dither only alpha values whose local quantization gap is ≥ 4 units (2026-07-11) | Adaptive level reduction can pack levels ~2 apart on low-range planes (shadow/overlay ramps spanning a fraction of 0–255). Banding that small is invisible, but the dither pattern still scrambles LZ77 — a personal-archive sweep found a 2.5× file (vs libwebp) that this rule alone brings to 1.6× with max alpha error 1. Gap-local, so a plane with both wide and narrow gaps dithers exactly where it can be seen. The same file exposed the filter-ranking estimator ignoring LZ77 mechanics — streaks under MIN_MATCH(4) are really literals, and every match pays a length+distance token — which once mis-ranked filters by 23%; `estimateVP8LCost` now models both. Fixture: `shade` in harness/content.ts |
| `effort` option | `"quality"` (default, RD + 4×4 modes) vs `"fast"` (prediction-error mode selection, no RD, no 4×4) (2026-07-11) | The RD/B_PRED encoder tripled encode time; as a WebKit gap-filler the default should give the best output (a canvas encode is one-shot and 15–25 ms at 256² is imperceptible), but bulk/interactive users get the old speed back explicitly. Measured on the corpus: fast is ~3× faster at nearly identical *average* size, paying ~1 dB average PSNR and a worse tail (p5 −1.37 vs −0.66), with up to ~40% larger files on gradient/edge content. Strictly validated (throws on unknown values) since no native contract covers it |
| Quantize vs filter order | Quantize **before** the prediction filter | Keeps residuals small and consistent |
| Mode search | Rate-distortion (SSE + λ·estimated coefficient cost), replacing the original SAD heuristics (2026-07-11) | The 5,229-file macOS-asset sweep showed the SAD encoder paying +29% VP8 bytes vs libwebp m3. RD alone cut that to +22% at *better* PSNR. λ runs at 1/16 of libwebp's 3q²: with single-pass default-probability rate estimates, full-strength λ amplified estimate noise into Pareto-worse picks — measured on smooth-content files that came out larger *and* ~2.5 dB worse than the SAD encoder until the rescale, after which λ/16 gave equal bytes at ~1.8 dB better PSNR on the calibration set. libwebp's flatness penalty deliberately not ported — measured as bytes-for-nothing at this operating point |
| 4×4 luma modes | **Added** (2026-07-11, reversing the 2026-07-02 gate decision) | The macOS-asset sweep overturned the old corpus verdict: banded-gradient/edge content (the V16 archetype) needed B_PRED to close a 1.87× size gap, and on icon-sized alpha content the sub-modes buy ~+0.7 dB for flat bytes. Full port of libwebp PickBestIntra4: per-block RD over the ten sub-modes, λ_mode arbiter vs the i16 score with early-out, per-MB header-bits cap, partition-0 overflow falls back to i16-only. Costs ~8 KB minified (mostly the 10×10×9 mode-probability table) |
| Adaptive token probabilities | Added during quality tuning | Cut corpus overhead 58.7% → 24.9%; ~120 lines, no decoder risk |
| Transparent-area handling | libwebp parity (1.1.0): 2×2 chroma averaging is gamma-corrected and, on mixed-alpha blocks, alpha-weighted; transparent luma is smoothed to the 8×8-block-local visible average and fully transparent 8×8 blocks are flattened (ports of `WebPAccumulateRGBA` + `WebPCleanupTransparentArea`, lossy path only) | The v1.0.0 global-mean fill let masked-out colors bleed into every alpha-edge chroma sample — the "washed-out thumbnails" gap vs Chromium (−2.6 dB blue at q0 on 16×16 thumbs). The port reaches metric-exact q0 parity with libwebp (`harness/probe.ts`). The lossless candidate still encodes untouched pixels so a lossless result round-trips exactly |
| Out-of-range `quality` | HTML canvas rule, scaled to 0–100: omitted, out-of-range, NaN, and ±Infinity all encode at the default 80 (2026-07-05) | slim is a WebKit gap-filler; throwing (or clamping) where every native `toBlob` silently substitutes its default would make the fallback path stricter than the path it replaces — an error surfacing only on Safari, the least-tested environment. slim-only options (`alphaDither` etc.) still validate strictly since no native contract covers them |
| Lossless candidate selection | Palette VP8L raced against lossy; smaller file wins (2026-07-02) | Amends the "no lossless RGB" non-goal for the ≤ 256-color subset only. Flat/sprite content comes out 2–6× smaller *and* pixel-perfect; obviates most of the deferred 4×4-modes work. Quality acts as a floor: lossless only replaces lossy when not larger |
| `quality: 100` semantics | Still lossy — no native-parity lossless switch (2026-07-04) | Browsers special-case `toBlob('image/webp', 1.0)` to full lossless VP8L; matching that requires general lossless RGB (predictor transform, color cache, real LZ77 — roughly +1,000 lines / +6–10 KB), a non-goal. Alpha already races a lossless plane and is typically exact; users needing pixel-exact output on WebKit have PNG |
| Rate control | None — fixed quality→QP | Thumbnails don't need target-size encoding |
| Bit-exactness reference | `cwebp -m 0` + instrumented diffs | Fastest debugging loop during bring-up |

**Note on the overhead figure:** the size comparison (`npm run compare`) disables both the lossless racing and alpha dithering to isolate the lossy VP8 core. Alpha dithering must stay out of this measurement: it lowers the composite SSIM without touching the VP8 chunk, which lets the matched-SSIM sweep pair us against a much lower cwebp quality and inflates the apparent overhead (verified 2026-07-04 — VP8 chunks byte-identical either way). As of the RD + B_PRED work (2026-07-11, incl. the `fade` corpus addition and the 32-level alpha default) this metric reads **−26%**, i.e. smaller than `cwebp -m 0` at matched SSIM; the honest reference is libwebp *method 3* via `harness/syseval.ts`, where slim's VP8 chunk runs ~+23% at ~+1 dB PSNR (whole-file totals land near parity with the banding-first 32-level alpha default, or several points smaller at `alphaLevels: 16`). History: 58.7% at bring-up → 24.9% with adaptive probabilities → −26% with RD+B_PRED.

## Bit-order hazard

The VP8L bit writer is **LSB-first**, the opposite of the VP8 boolean coder. Both are covered by isolated round-trip unit tests (`harness/booltest.ts`, `harness/vp8ltest.ts`) precisely because silent corruption here was the largest development risk.
